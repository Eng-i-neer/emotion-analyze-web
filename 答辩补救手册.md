## 二、项目一句话介绍

**1. 简短版（10秒）：**

"这是一个个人情绪记录网站，用户写日记，AI 自动分析情绪状态并给出建议。"

**2. 正式版（20秒）：**

"这是一个基于 Flask 和 DeepSeek AI 的个人情绪管理网站。用户每天写文字日记，系统调用 AI 对日记内容进行情绪分析，自动识别情绪类型、压力来源和关键词，生成评分和调节建议，并通过趋势图、日历、周报等形式帮助用户了解自己的情绪变化。"

**3. 老师问"你这个项目到底解决什么问题"时的回答：**

"很多人每天都有情绪波动，但自己不太善于察觉和管理。传统日记只是记录文字，没有反馈。我们这个系统让用户写完日记后，AI 帮忙'读懂'这段文字，告诉用户今天主要情绪是什么、压力大不大、有什么值得关注的，还会给温和的调节建议。时间长了，用户还可以看趋势图和月报，了解自己的情绪变化规律。它不是心理诊断工具，而是一个帮助自我观察的情绪记录助手。"

## 三、系统整体架构怎么讲

### 用大白话讲架构：

"我们这个系统分四层：

1. **前端**就是用户看到的网页界面，用 HTML、CSS、JavaScript 写的。用户在这里写日记、看图表、看分析结果。

2. **后端**是服务器端的程序，用 Python 的 Flask 框架写的。它负责接收前端提交的数据、处理业务逻辑、调用 AI、操作数据库。可以理解为'后台管家'。

3. **数据库**用的 SQLite，就是一个本地数据库文件，存用户信息、日记记录、AI 分析结果、报告。

4. **AI 接口**调的是 DeepSeek 大语言模型的 API。我们把日记文本发给 AI，AI 返回结构化的分析结果。"

### 一句话版本数据流（背下来）：

"用户在前端写完日记并提交，前端通过 API 把数据发给后端，后端先把原始日记存到数据库，然后调用 DeepSeek AI 分析日记文本，AI 返回一个 JSON 格式的分析结果，后端再把分析结果更新回数据库，最后前端读取这些数据展示成分数卡片、标签、图表和建议。"

## 四、老师常问技术问题与回答

### 1. 你这个网站用了什么技术栈？

**老师可能怎么问：** "说说你们用了什么技术？"

**回答：** "后端用的 Python Flask 框架，数据库用 SQLite，前端是原生 HTML + CSS + JavaScript 做的单页应用，AI 分析调用的是 DeepSeek 大语言模型的 API，图表是用 Canvas 手动绘制的。"

**追问补充：** "选 Flask 是因为它轻量，适合课程设计规模的项目。SQLite 不需要额外装数据库服务器，部署简单。前端没用 Vue 或 React 是因为对课程设计来说原生 JS 足够了，而且能更好地理解底层逻辑。"

---

### 2. 前端负责什么？

**老师可能怎么问：** "前端做了些什么？"

**回答：** "前端负责所有用户看到的界面。包括登录注册页面、日记编辑器、AI 分析结果展示、历史记录列表、情绪日历、数据分析图表、周报月报展示、AI 建议页面。前端通过 fetch 请求和后端 API 通信，所有页面切换在一个 HTML 文件里完成，是单页应用的模式。"

**追问补充：** "图表我们没有用 ECharts 之类的第三方库，是用 Canvas API 手动画的折线图、饼图、柱状图、散点图，这样更轻量，也能更好地控制样式。"

---

### 3. 后端负责什么？

**老师可能怎么问：** "后端的主要功能是什么？"

**回答：** "后端主要做五件事：一是用户认证，处理注册登录；二是日记记录的增删改查；三是调用 AI 接口分析日记；四是统计数据的聚合，比如计算平均分、趋势数据；五是生成周报月报。后端用 Flask 的 Blueprint 把这些功能分成了 auth、mood、ai、stats、report 五个模块，每个模块有独立的路由文件和业务逻辑文件。"

**追问补充：** "这种分模块的方式叫做分层架构，路由层负责接收请求，服务层负责业务逻辑，模型层负责数据库操作，各层职责分明。"

---

### 4. 数据库里存了什么？

**老师可能怎么问：** "数据库有哪些表？分别存什么？"

**回答：** "数据库有三张主表。users 表存用户名和加密后的密码。mood_records 表是最核心的，存每条日记的原始文本、日期、以及 AI 分析后的所有字段，包括情绪评分、压力评分、精力评分、主要情绪、关键词、标签、建议等等，还有一个字段存了 AI 返回的完整 JSON。reports 表存周报月报的汇总数据和 AI 生成的报告内容。"

**追问补充：** "mood_records 表上建了一个联合索引 (user_id, record_date)，这样按用户和日期查询会更快。"

---

### 5. AI 是怎么分析日记的？

**老师可能怎么问：** "AI 分析的原理是什么？"

**回答：** "我们调用的是 DeepSeek 大语言模型的 API。具体来说，后端会构造一个请求，包含两部分：一个是系统提示词，告诉 AI 它的角色是'情绪观察助手'，规定了它需要返回哪些字段、什么格式、不能做医学诊断等规则；另一个是用户的日记内容。AI 理解日记文本后，返回一个结构化的 JSON，里面包含情绪评分、压力评分、主要情绪、关键词、建议等等。我们用了 response_format 参数强制 AI 返回 JSON 格式，避免它返回自由文本。"

**追问补充：** "这种方式叫做 Prompt Engineering，就是通过精心设计提示词来控制 AI 的输出。我们的系统提示词大概有 30 多行，定义了每个字段的含义和取值范围。"

---

### 6. 为什么不用传统关键词匹配，而用 AI？

**老师可能怎么问：** "为什么不用规则或关键词匹配？"

**回答：** "关键词匹配只能识别'开心''难过'这样的明确词汇，但很多时候用户不会直接写'我很焦虑'，而是写'最近总感觉做什么都提不起劲'。大语言模型可以理解上下文和隐含语义，能分析出这段话的实际情绪是什么。而且 AI 可以同时做多维度分析，一次返回情绪、压力、关键词、建议，关键词匹配很难做到这种综合判断。"

**追问补充：** "当然 AI 也不是百分百准确，所以我们的定位是辅助自我观察，不是诊断工具。"

---

### 7. AI 返回的数据格式是什么？

**老师可能怎么问：** "AI 返回什么格式的数据？"

**回答：** "AI 返回的是一个 JSON 对象，包含十几个字段。比如 summary 是一句话总结、main_emotion 是主要情绪、emotion_score 是 1-10 的情绪评分、stress_score 是压力评分、keywords 是关键词列表、suggestions 是建议列表、risk_level 是关注等级。这些字段都是在系统提示词里预先定义好的。"

**追问补充：** "我们在代码里还做了兜底处理，如果 AI 返回的 JSON 缺少某个字段，会自动用默认值补全。"

---

### 8. 什么是 JSON？你这里为什么用 JSON？

**老师可能怎么问：** "你一直说 JSON，解释一下什么是 JSON？"

**回答：** "JSON 是一种轻量级的数据交换格式，它用键值对的方式组织数据，比如 {\"name\": \"张三\", \"score\": 8}。我们项目里用 JSON 主要有三个地方：一是前后端通信，前端提交数据和后端返回结果都用 JSON；二是 AI 接口的输入输出也是 JSON；三是数据库里存 AI 分析结果时，也把完整的 JSON 存了一份，方便前端直接读取展示。"

**追问补充：** "JSON 的好处是人能看懂，程序也好解析，几乎所有编程语言都支持。"

---

### 9. 图表数据从哪里来？

**老师可能怎么问：** "这些图表的数据怎么获取的？"

**回答：** "图表数据来自后端的统计接口。后端有一个 stats 模块，会从 mood_records 表里按日期、按用户进行 SQL 聚合查询，比如计算每天的平均情绪分、每天的平均压力分，然后返回给前端。前端拿到数据后用 Canvas API 画出折线图、饼图等。不是每次都要重新调 AI，而是直接查数据库里已经存好的分析结果。"

**追问补充：** "比如情绪趋势图，后端的 SQL 大概是 SELECT record_date, AVG(ai_emotion_score) FROM mood_records WHERE user_id=? GROUP BY record_date，就是按日期分组取平均值。"

---

### 10. 情绪趋势图是怎么生成的？

**老师可能怎么问：** "情绪趋势图怎么实现的？"

**回答：** "前端请求后端的 /api/stats/trends 接口，传入天数参数比如 30 天，后端从数据库查出这 30 天内每天的情绪均分、压力均分、精力均分，返回一个数组。前端拿到数据后，用 JavaScript 的 Canvas API 手动绘制折线图，包括坐标轴、网格线、数据点、填充渐变区域。"

**追问补充：** "我们没用 ECharts 第三方库，是手写的 Canvas 绑制逻辑，这样图表风格能和我们的手帐 UI 更统一。"

---

### 11. 情绪日历是怎么生成的？

**老师可能怎么问：** "日历上的表情是怎么来的？"

**回答：** "后端有一个 calendar 接口，传入年份和月份，查出当月每天的平均情绪分。前端拿到数据后，根据分值范围映射成不同的表情：8分以上显示😊，5-7分显示😐，3-4分显示😟，3分以下显示😢。没有记录的日子就显示日期数字，有记录的日子会有一个小彩色圆点。"

**追问补充：** "点击某一天还可以展开当天的记录详情。"

---

### 12. 周报/月报怎么生成？

**老师可能怎么问：** "周报月报是怎么自动生成的？"

**回答：** "用户点击'生成周报'后，后端先从数据库取出最近一周的所有记录，把每条记录的关键信息汇总成一段文字摘要，包括日期、情绪、压力、关键词等。然后把这段摘要发给 AI，让 AI 生成一份结构化的周期报告，包含整体总结、情绪趋势描述、压力趋势描述、积极变化、需要关注的点和下阶段建议。后端再计算平均分、最低情绪日等统计数据，把所有内容合在一起存入 reports 表。"

**追问补充：** "月报也是同样的逻辑，只是时间范围改为当月。"

---

### 13. 用户写一篇日记之后，系统内部发生了什么？

**老师可能怎么问：** "从用户写日记到看到分析结果，中间发生了什么？"

**回答：** "一共分六步：第一步，用户在前端写完日记点'提交并分析'；第二步，前端把日记文本和日期通过 POST 请求发给后端；第三步，后端把原始日记存到 mood_records 表，拿到记录 ID；第四步，后端把这条日记的文本发给 DeepSeek AI 接口；第五步，AI 返回 JSON 分析结果后，后端把所有分析字段更新回 mood_records 表；第六步，后端把完整的记录返回给前端，前端渲染分析卡片，显示情绪分、压力分、关键词、建议等。"

**追问补充：** "整个过程大概几秒钟，其中主要的等待时间是 AI 分析那一步。"

---

### 14. 如果 AI 接口调用失败怎么办？

**老师可能怎么问：** "AI 挂了怎么办？系统会崩吗？"

**回答：** "不会崩。我们的代码里做了异常处理，AI 调用失败时会 catch 异常，然后返回一套默认值。默认值包括情绪分 5 分、压力分 5 分、主要情绪'未知'、建议'记录本身就是一种自我关照'。这样即使 AI 不可用，用户的日记也已经保存了，只是缺少 AI 分析。等 AI 恢复后可以重新分析。"

**追问补充：** "这种设计叫做'优雅降级'，就是核心功能不受第三方服务故障影响。"

---

### 15. 如果用户日记很短，AI 怎么分析？

**老师可能怎么问：** "用户只写了'今天心情不好'这几个字，AI 能分析吗？"

**回答：** "能。我们在系统提示词里专门写了'如果文本很短也要尽量分析'。AI 会根据有限的文字给出保守的分析结果。当然文字越多，分析就越准确。短文本的分析结果可能比较笼统，但至少能识别出基本的情绪倾向，比如这几个字 AI 能判断出主要情绪是'难过'或者'低落'。"

**追问补充：** "这就像医生问诊一样，你说得越详细，判断越准确。"

---

### 16. 如果 AI 分析不准怎么办？

**老师可能怎么问：** "AI 分析结果可靠吗？不准怎么办？"

**回答：** "首先我们的定位不是诊断工具，而是辅助自我观察。AI 的分析是一个参考，帮助用户从另一个角度理解自己的文字。其次，我们把所有分析结果都存了下来，用户可以在历史记录里回顾。长期积累的数据趋势比单次分析更有意义，因为偶尔的偏差会在整体趋势中被平滑掉。最后，系统还保留了完整的日记原文，用户随时可以对照 AI 的分析和自己的实际感受。"

**追问补充：** "就像体温计偶尔有误差，但连续测一周的趋势是有参考价值的。"

---

### 17. 你怎么避免 AI 做心理疾病诊断？

**老师可能怎么问：** "AI 会不会给用户诊断抑郁症什么的？"

**回答：** "不会。我们在系统提示词里明确写了两条限制：第一，'不要做医学诊断，不要提及任何心理疾病名称'；第二，risk_level 只是日常关注等级，normal 是正常、watch 是需要留意、high 是建议寻求支持，不是疾病诊断。AI 的建议也被限制为生活化的、温和的、像朋友关心一样的内容，比如'试着去散散步'，而不是'你可能有焦虑症'。"

**追问补充：** "如果 AI 检测到极端内容，会在建议里温和地提示用户联系信任的人或专业支持，但不会做任何诊断。"

---

### 18. risk_level 是不是心理疾病判断？

**老师可能怎么问：** "这个风险等级是什么意思？是在判断心理疾病吗？"

**回答：** "不是。risk_level 翻译过来是'关注等级'，不是医学风险评估。normal 表示状态正常、watch 表示最近可能需要多关注一下自己、high 表示建议用户找信任的人聊聊或寻求专业支持。这更像是一个'关心提醒'，类似手机提醒你该喝水了，而不是说你生病了。"

**追问补充：** "我们特意避免用'危险''高风险'这样的措辞，用的是'留意观察''建议寻求支持'，保持温和。"

---

### 19. 这个系统和普通日记本有什么区别？

**老师可能怎么问：** "用户直接用备忘录写日记不就行了？你这个多了什么？"

**回答：** "普通日记本只是记录，写完就放在那里了。我们这个系统多了三层价值：第一层是 AI 即时反馈，写完就能看到情绪分析和建议；第二层是数据积累后的可视化，情绪趋势图、情绪日历能让用户看到长期的情绪变化规律；第三层是周期性报告，AI 会总结一周或一个月的整体状态，帮用户发现自己可能没注意到的模式。"

**追问补充：** "简单说就是从'记录'升级到了'记录 + 理解 + 反馈'。"

---

### 20. 这个项目的创新点是什么？

**老师可能怎么问：** "你觉得这个项目有什么创新？"

**回答：** "主要有三个创新点：第一，用 AI 替代用户自评。很多人不善于评价自己的情绪，我们让 AI 读日记后自动给出情绪评分、压力等级、情绪类型等，比用户自己打分更客观。第二，AI 的分析结果是结构化的，不是一段模糊的文字，而是明确的分数、标签、关键词，这样才能用于统计和可视化。第三，强调了'非诊断'定位，从系统提示词层面就限制了 AI 不能做医学诊断，这在类似项目中是需要特别设计的。"

---

### 21. 这个项目的难点是什么？

**老师可能怎么问：** "做这个项目最难的是什么？"

**回答：** "主要有三个难点：第一是 AI 输出的可控性，大语言模型有时候返回的格式不规范，我们需要设计详细的系统提示词、用 JSON 模式强制格式、并且做异常处理和默认值兜底。第二是数据的多维度设计，一条日记要关联十几个分析字段，数据库表设计和前端展示都需要仔细规划。第三是图表的手动实现，我们没用第三方图表库，用 Canvas API 从头画折线图、饼图、柱状图，需要处理坐标计算、设备像素比、响应式等问题。"

---

### 22. 数据库表是怎么设计的？

**老师可能怎么问：** "数据库有几张表？字段怎么设计的？"

**回答：** "三张表。users 表很简单，存 id、用户名、密码哈希和注册时间。mood_records 是核心表，有二十多个字段，前半部分是用户输入的原始数据比如日期和日记文本，后半部分是 AI 分析结果比如 ai_emotion_score、ai_main_emotion、ai_keywords 等，还有一个 ai_analysis_json 字段存了 AI 返回的完整 JSON。reports 表存周报月报，包含类型、时间范围、平均分和 AI 生成的报告 JSON。"

**追问补充：** "数组类型的字段比如 keywords、suggestions，在 SQLite 里是用 TEXT 类型存 JSON 字符串，读出来的时候用 json.loads 解析。"

---

### 23. 用户数据怎么保证隐私？

**老师可能怎么问：** "日记这么隐私的内容，安全吗？"

**回答：** "我们做了几层保护。第一，用户必须登录才能访问数据，所有 API 都有登录检查。第二，数据隔离，每个用户只能查看和操作自己的记录，SQL 查询里都带了 user_id 条件。第三，密码不是明文存储，用的 werkzeug 库的密码哈希函数。第四，Session Cookie 设置了 HttpOnly 和 SameSite=Lax，防止 XSS 和 CSRF 攻击。当然，目前是课程设计版本，如果部署到线上还需要加 HTTPS、数据加密等更多安全措施。"

---

### 24. 密码怎么保存？

**老师可能怎么问：** "密码是明文存的吗？"

**回答：** "不是。我们用的 werkzeug 库的 generate_password_hash 函数对密码做哈希处理后再存入数据库。登录时用 check_password_hash 比对，整个过程不会存储也不会传输明文密码。就算数据库被泄露，攻击者拿到的也是哈希值，很难反推出原密码。"

**追问补充：** "werkzeug 默认用的是 pbkdf2 加盐哈希，每个用户的盐值不同。"

---

### 25. 多用户之间的数据怎么隔离？

**老师可能怎么问：** "A 用户能看到 B 用户的日记吗？"

**回答：** "不能。所有数据操作都通过后端 API 进行，后端会从 Session 里读取当前登录用户的 user_id，然后所有数据库查询都带上 WHERE user_id=? 的条件。所以每个用户只能看到和操作自己的数据。"

---

### 26. 为什么要把 AI 分析结果存数据库，而不是每次重新分析？

**老师可能怎么问：** "为什么要存 AI 的结果？每次查的时候重新分析不行吗？"

**回答：** "有三个原因。第一，省钱，AI 接口调用是有成本的，每次看历史记录都调一次不现实。第二，保证一致性，同一篇日记每次调 AI 可能返回不同结果，存下来就能保证用户看到的分析是稳定的。第三，支持统计，趋势图、日历、周报都需要历史数据的聚合查询，直接从数据库取比每次调 AI 快得多。"

---

### 27. 前后端接口怎么设计？

**老师可能怎么问：** "你们的 API 是怎么设计的？"

**回答：** "我们用了 RESTful 风格的 API 设计。按功能分成了五组：/api/auth 处理认证、/api/mood-records 处理日记的增删改查、/api/ai 处理 AI 分析和建议、/api/stats 提供统计数据、/api/reports 处理周报月报。每个接口用不同的 HTTP 方法区分操作，比如 POST 创建、GET 查询、PUT 修改、DELETE 删除。所有接口返回统一的 JSON 格式，包含 success 和 data 字段。"

### 28. 什么是 API？

**老师可能怎么问：** "你说的 API 是什么意思？"

**回答：** "API 就是程序之间通信的接口。在我们项目里有两层 API：一层是我们自己写的后端接口，前端通过这些接口向后端发送和获取数据；另一层是 DeepSeek 的 AI 接口，我们的后端通过它调用 AI 模型做文本分析。可以理解为 API 就像餐厅的菜单，你按菜单点菜，厨房按菜单做菜，不需要知道厨房内部怎么运作。"

### 1. 用户情绪记录流程

```
用户打开"写日记"页面
      ↓
在方格纸编辑器中写下当天的感受
      ↓

      ↓
前端发送 POST 请求到 /api/mood-records
      ↓
后端接收数据，存入 mood_records 表（原始日记）
      ↓
后端返回记录 ID
      ↓
前端发送 POST 请求到 /api/ai/analyze-diary，带记录 ID
      ↓
后端取出日记文本，调用 DeepSeek AI 接口
      ↓
AI 返回结构化 JSON 分析结果
      ↓
后端将分析结果更新回 mood_records 表
      ↓
后端返回完整记录给前端
      ↓
前端渲染 AI 分析卡片（情绪分、压力分、关键词、建议等）
```

### 2. AI 分析流程

```
后端构造请求：
  ├── System Prompt（角色定义 + 字段规范 + 限制条件）
  └── User Message（"日记内容：xxx"）
      ↓
发送到 DeepSeek API（response_format: json_object）
      ↓
AI 理解文本语义
      ↓
返回 JSON：
  ├── summary（一句话总结）
  ├── main_emotion（主要情绪）
  ├── emotion_score / stress_score / energy_score
  ├── keywords / auto_tags / stress_sources
  ├── risk_level / suggestions
  └── 其他辅助字段
      ↓
后端校验 JSON 完整性，缺失字段用默认值补全
      ↓
写入数据库
```

### 3. 历史记录展示流程

```
用户打开"历史日记"页面
      ↓
前端发送 GET 请求到 /api/mood-records（可带筛选参数）
      ↓
后端根据 user_id + 筛选条件查询数据库
      ↓
返回记录列表（含日期、摘要、情绪分等）
      ↓
前端渲染记录列表卡片
      ↓
用户点击某条记录
      ↓
前端请求 /api/mood-records/{id} 获取详情
      ↓
弹窗展示完整分析结果 + 日记原文
```

### 4. 情绪趋势图生成流程

```
用户打开"数据分析"页面
      ↓
前端请求 /api/stats/trends?days=30
      ↓
后端 SQL 聚合：
  SELECT record_date, AVG(ai_emotion_score), AVG(ai_stress_score)...
  FROM mood_records WHERE user_id=? GROUP BY record_date
      ↓
返回每天的平均分数组
      ↓
前端用 Canvas API 绘制折线图
  ├── 计算坐标映射

```
用户点击"生成周报"
      ↓

      ↓

      ↓

```

### 6. 异常处理流程

```
AI 接口调用
      ↓

## 六、数据库表怎么讲

### 老师问数据库设计时的回答话术：

"我们数据库用的 SQLite，总共三张表：

**users 表**存用户账号信息，字段有 id（自增主键）、username（唯一）、password_hash（密码哈希值，不存明文）、created_at（注册时间）。

**mood_records 表**是核心表，二十多个字段。前面几个是基础字段：user_id 关联用户、record_date 日期、diary_text 日记原文。中间是 AI 分析字段：ai_summary 总结、ai_main_emotion 主要情绪、ai_emotion_score 情绪分、ai_stress_score 压力分、ai_energy_score 精力分、ai_keywords 关键词、ai_suggestions 建议列表等。最后还有一个 ai_analysis_json 字段存了 AI 返回的完整 JSON，方便前端整体读取。数组类型的字段比如关键词、标签，都是用 TEXT 存 JSON 字符串。

**reports 表**存周报月报，包括报告类型（周报/月报）、时间范围、平均情绪分、平均压力分、以及 AI 生成的报告 JSON。

表之间通过 user_id 做外键关联。mood_records 表上建了 (user_id, record_date) 的联合索引加快查询。"

## 八、项目亮点怎么讲

### 亮点 1：AI 文本理解与多维度分析

"我们不是让用户自己打分，而是让 AI 读懂日记文本后自动给出情绪评分、压力评分、精力评分、情绪类型、关键词、标签和建议。一次分析出十几个维度，这是传统自评量表做不到的。"

### 亮点 2：情绪数据可视化

"我们用 Canvas API 手写了六种图表：情绪趋势折线图、压力趋势折线图、精力趋势折线图、睡眠与情绪的散点图、情绪类型饼图、标签统计柱状图。让数据不只是冷冰冰的数字，而是直观的视觉趋势。"

### 亮点 3：情绪日历

"情绪日历把每天的情绪用表情符号展示在月历上，用户一眼就能看到这个月哪些天心情好、哪些天不太好。比传统的日历多了情感维度。"

### 亮点 4：AI 生成周报/月报

"系统会自动汇总一段时间内的记录，发给 AI 生成周期性报告，包含情绪趋势总结、积极变化、需要关注的点和下阶段建议。帮助用户从更长的时间维度理解自己。"

### 亮点 5：严格的非医疗定位

"我们从设计之初就明确了不做心理诊断。在系统提示词里禁止了所有医学诊断内容，risk_level 只是关注等级不是疾病风险，建议都是生活化的温和建议。这是一个伦理层面的设计考量。"

## 九、项目不足怎么讲

### 不足 1：AI 分析准确性

"AI 分析的准确性依赖模型能力和日记文本质量，短文本或表述模糊时准确率会降低。"
→ **改进方向：** "可以考虑让用户对 AI 分析结果做反馈，比如'准确/不准确'，积累数据后优化提示词。"

### 不足 2：数据量有限

"目前只是个人维度的数据分析，数据量小的时候趋势图参考价值有限。"
→ **改进方向：** "持续使用积累数据后会越来越有价值，也可以加入基准参考线。"

### 不足 3：不做专业心理诊断

"系统只做日常自我观察，不能替代专业心理咨询和诊断。"
→ **改进方向：** "可以接入专业心理健康资源链接，在需要时引导用户寻求专业帮助。"

### 不足 4：隐私安全基础

"目前是课程设计版本，安全措施比较基础，密码哈希和 Session 认证是有的，但没有 HTTPS 和数据加密。"

## 十二、老师连续追问时如何应对

### 场景 1：AI 分析原理追问

**老师：** "AI 是怎么分析日记的？"

**我：** "后端把日记文本和一段系统提示词一起发给 DeepSeek 的 API。系统提示词定义了 AI 的角色是情绪观察助手，规定了它需要返回哪些字段、什么格式、不能做诊断。AI 理解文本后返回一个 JSON，包含情绪分、压力分、关键词、建议等。"

**老师：** "那 AI 怎么知道用户是焦虑还是开心？"

**我：** "AI 是大语言模型，它在预训练阶段学习了大量的文本，理解了不同词汇和句式所表达的情感。比如用户写'做什么都提不起劲'，AI 能从语义上理解这是一种低落或疲惫的表达。它不是靠简单的关键词匹配，而是理解整段话的语境和含义。"

**老师：** "那如果 AI 判断错了呢？"

**我：** "有可能出现偏差，所以我们的定位是辅助参考而不是诊断。我们保留了日记原文让用户可以对照。而且长期积累的趋势数据比单次分析更可靠，偶尔的误差会在整体趋势中被平滑。后续也可以加入用户反馈机制来优化。"

---

### 场景 2：数据库设计追问

**老师：** "你说数据库存了 AI 分析结果，具体存了什么？"

**我：** "mood_records 表里有十几个 AI 相关字段，比如 ai_summary 存一句话总结、ai_emotion_score 存 1-10 的情绪分、ai_main_emotion 存主要情绪、ai_keywords 存关键词数组、ai_suggestions 存建议数组。数组类型的字段用 TEXT 类型存 JSON 字符串。另外还有一个 ai_analysis_json 字段存了 AI 返回的完整 JSON。"

**老师：** "为什么既单独存字段又存完整 JSON？不是冗余吗？"

**我：** "确实有一定冗余，但这样设计有好处。单独的字段方便做 SQL 查询和聚合统计，比如计算平均情绪分只需要 AVG(ai_emotion_score)。完整的 JSON 则方便前端直接读取展示，不用从多个字段拼装。在课程设计的数据量下，这点冗余完全可以接受。"

**老师：** "为什么用 SQLite 而不用 MySQL？"

**我：** "SQLite 是文件级数据库，不需要单独安装数据库服务器，一个 .db 文件就是整个数据库，部署非常简单。对于课程设计这种单机运行、用户量小的场景完全够用。如果后续需要支持高并发多用户，可以迁移到 MySQL 或 PostgreSQL。"

---

### 场景 3：安全性追问

**老师：** "这个系统安全吗？"

**我：** "做了基础的安全措施。密码用 werkzeug 库做了哈希存储，不存明文。用户认证用 Flask 的 Session 机制，Cookie 设置了 HttpOnly 和 SameSite。所有 API 都做了登录检查，数据按 user_id 隔离。"

**老师：** "如果有人抓包怎么办？"

**我：** "目前开发环境是 HTTP，数据确实是明文传输的。生产部署时需要配置 HTTPS 来加密传输层。这属于运维层面的安全配置，课程设计阶段我们先关注功能逻辑的安全性。"

**老师：** "有没有防 SQL 注入？"

**我：** "有。我们所有的 SQL 操作都用了参数化查询，就是用问号占位符传参，而不是字符串拼接。这是 SQLite3 库推荐的写法，能有效防止 SQL 注入。"

---

### 场景 4：前端技术追问

**老师：** "前端为什么不用 Vue 或 React？"

**我：** "主要是课程设计的定位，用原生 JS 能更好地理解底层原理，比如 DOM 操作、事件处理、Canvas 绑制。而且我们的页面数量不算多，用框架反而增加了学习成本和项目复杂度。"

**老师：** "图表为什么不用 ECharts？"

**我：** "两个原因。一是想锻炼 Canvas API 的使用能力，手写图表更能理解坐标映射、数据可视化的底层原理。二是第三方图表库的默认风格比较商务化，和我们手帐风格的 UI 不太搭，手写可以完全控制视觉风格。"

**老师：** "单页应用是怎么实现的？"

**我：** "一个 index.html 里放了所有页面的容器 div，一次只显示一个。页面切换时通过 JavaScript 控制哪个 div 显示哪个隐藏，同时调用对应页面的初始化函数加载数据。导航栏点击不会刷新整个页面，这就是单页应用的基本原理。"

### 场景 5：项目价值追问

**老师：** "这个项目有什么实际用处？"

## 十四、5 分钟项目介绍稿

大家好，我今天汇报的项目是**个人情绪记录与 AI 心理状态分析网站**。

**一、项目背景**

心理健康越来越受到关注，但大多数人不会主动做心理咨询。传统的情绪记录方式，比如写日记或做量表，要么没有反馈，要么操作负担重。我们希望做一个低门槛的情绪管理工具：用户只需要像写手帐一样记录每天的感受，AI 自动帮忙分析和反馈。

**二、系统架构**

系统分为四层。前端是原生 HTML、CSS、JavaScript 构建的单页应用，风格是手帐笔记本风，方格纸背景、装订环、胶带效果。后端用 Python Flask 框架，分为认证、记录、AI分析、统计、报告五个模块。数据库用 SQLite，三张主表：用户表、情绪记录表和报告表。AI 接口调用的是 DeepSeek 大语言模型。

**三、核心功能**

第一个核心功能是**日记记录与 AI 分析**。用户在方格纸编辑器里写日记，点"提交并分析"后，系统先保存日记，再调用 AI 分析。AI 会返回结构化的 JSON，包括一句话总结、主要情绪、次要情绪、情绪评分 1-10 分、压力评分、精力评分、关键词、自动标签、压力来源、关注等级和调节建议。这些数据全部存入数据库，并在前端以卡片形式展示。

第二个核心功能是**数据可视化**。包括情绪趋势折线图、压力趋势折线图、睡眠与情绪的散点图、情绪类型饼图、标签统计柱状图等。这些图表用 Canvas API 手动绘制，数据来自后端的聚合查询。

第三个核心功能是**情绪日历**。以月历形式展示每天的情绪状态，用表情符号表示：😊表示心情好、😐表示一般、😟表示偏低、😢表示很差。点击某一天可以查看当天的详细记录。

第四个核心功能是**AI 周报月报**。系统汇总一段时间内的记录发给 AI，生成周期性报告，包含整体总结、情绪趋势、压力趋势、积极变化、需要关注的问题和下阶段建议。

**四、AI 分析设计**

我们通过精心设计的系统提示词来控制 AI 的输出。提示词定义了 AI 的角色、返回字段、取值范围，并明确禁止做医学诊断。使用 response_format 参数强制 JSON 输出。代码层面做了异常处理，AI 接口失败时返回默认值，保证系统不会因为 AI 问题而崩溃。

**五、安全设计**

密码用哈希存储，Session 认证，Cookie 设置了 HttpOnly 和 SameSite，数据按用户隔离。SQL 操作用参数化查询防注入。

**六、项目亮点**

一是 AI 替代用户自评，降低了使用门槛；二是结构化的分析结果支持统计和可视化；三是严格的非医疗定位，从技术层面限制了 AI 的输出边界。

**七、不足与改进**

AI 准确性依赖模型能力和文本质量，后续可加入用户反馈优化。安全措施比较基础，生产环境需要 HTTPS 和数据加密。移动端体验可以继续优化。

总的来说，这个项目验证了"AI + 日记"做日常情绪管理的可行性，把记录、分析、可视化、报告整合在了一个系统里。谢谢大家。
